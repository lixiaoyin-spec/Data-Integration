|------|------|------|
| POST | `/api/drop` | **跨院退选**：验证参数 → 检查目标学院在线 → 调用目标学院 `POST /x/drop` |

## 三、跨院退选模块

### 3.1 请求与响应

**请求体** (JSON 或 form-urlencoded):
```json
{
  "sno": "S001",
  "cno": "C001",
  "target_college": "B"
}
```

**成功响应:**
```json
{
  "status": "success",
  "message": "退课成功"
}
```

**失败响应示例:**

| 场景 | HTTP | message |
|------|------|---------|
| 缺少参数 | 400 | 缺少必要参数: sno, cno, target_college |
| 无效学院 | 400 | 无效的学院 |
| 学院离线 | 503 | 学院B服务器不可用 |
| 目标学院退课失败 | 400 | 退课失败: … |
| 服务器异常 | 500 | 退课失败: … |

### 3.2 处理流程

```
POST /api/drop
{ sno, cno, target_college }
         │
         ▼
┌─────────────────────────────────────┐
│ Step 1: 解析 JSON / form 参数        │
│         target_college 自动转大写    │
├─────────────────────────────────────┤
│ Step 2: 校验 sno、cno、target_college│
├─────────────────────────────────────┤
│ Step 3: registry.get_client(target)  │
├─────────────────────────────────────┤
│ Step 4: client.check_status()        │
│         离线 → 503 返回              │
├─────────────────────────────────────┤
│ Step 5: client.drop(sno, cno)        │
│         → POST /x/drop               │
│         body: { sid, cid }           │
├─────────────────────────────────────┤
│ 返回: { status, message }            │
└─────────────────────────────────────┘
```

### 3.3 与选课模块的关系

| 对比项 | 跨院选课 `/api/enroll` | 跨院退选 `/api/drop` |
|--------|------------------------|----------------------|
| 复杂度 | 多步（导入学生 + 选课） | 单步转发 |
| 目标 | 目标学院创建选课记录 | 目标学院删除选课记录 |
| 参数 | sno, snm, sex, sde, source_college, cno, cnm, target_college | sno, cno, target_college |
| 学院客户端 | `import_student()` + `enroll()` | `drop()` |

退选模块设计为**轻量转发**：集成层不维护本地选课副本，直接委托课程所属学院（`target_college`）执行退课，与成员3选课写回形成对称操作。

## 四、全局统计模块

### 4.1 响应格式

```json
{
  "students_total": 150,
  "courses_total": 30,
  "enrollments_total": 750,
  "details": {
    "A": {
      "college_id": "A",
      "college_name": "学院A",
      "dbms": "SQL Server",
      "online": true,
      "students": 50,
      "courses": 10,
      "enrollments": 250
    },
    "B": { "...": "..." },
    "C": {
      "college_id": "C",
      "college_name": "学院C",
      "dbms": "MySQL",
      "online": false,
      "students": 0,
      "courses": 0,
      "enrollments": 0,
      "error": "无法连接"
    }
  }
}
```

### 4.2 统计流程

```
GET /api/statistics
         │
         ▼
┌──────────────────────────────────────────────────────────┐
│ for client in registry.all():                             │
│                                                           │
│   ┌─ check_status() ─ 离线?                               │
│   │     → details[cid] = { online:false, error, 0/0/0 }  │
│   │     → continue（不影响其他学院）                       │
│   │                                                       │
│   ├─ get_students()   → GET /x/students                  │
│   │     → parse_students_xml() → students 计数            │
│   │                                                       │
│   ├─ get_courses()    → GET /x/courses                     │
│   │     → parse_courses_xml() → courses 计数              │
│   │                                                       │
│   └─ get_enrollments() → GET /x/selections/xml            │
│         → count_enrollments_xml() → enrollments 计数      │
│                                                           │
│   累加 students_total / courses_total / enrollments_total │
└──────────────────────────────────────────────────────────┘
         │
         ▼
返回 JSON（含全局汇总 + details 字典）
```

### 4.3 异构选课 XML 兼容

三学院选课 XML 根元素与记录标签不同，`count_enrollments_xml()` 按优先级识别：

| 学院 | XML 结构 | 识别标签 |
|------|----------|----------|
| A (SQL Server) | `<Choices><choice>…` | `choice` |
| B (Oracle) | `<Selections><selection>…` | `selection` |
| C (MySQL) | `<Choices><choice>…` 或 enrollment 交换格式 | `choice` / `enrollment` |

### 4.4 容错策略

- **单学院离线**：该学院 `online: false`，计入 `details` 但不参与总数累加，全局接口仍返回 200
- **单学院部分数据失败**：已获取的数据仍统计，未获取项为 0，`error` 字段记录原因（如「获取选课数据失败」）
- **单学院异常**：`try/except` 捕获，`details[cid].error` 写入异常信息，继续处理下一学院
- **全局异常**：返回已收集的部分数据 + 顶层 `error` 字段，HTTP 500

## 八、其他成员依赖本模块的方式

### 成员1（集成服务器核心）
- 退选、统计逻辑已写入 `main.py`，与成员1的路由框架、`CollegeRegistry` 无缝集成
- 复用成员1的 `parse_students_xml()`、`parse_courses_xml()`，无需重复解析逻辑

### 成员2（课程聚合 + XSLT）
- 统计模块直接拉取各学院原始 XML 计数，与 `CourseAggregator` 去重聚合互补
- 成员2的 `courses_to_summary.xsl` 可用于课程维度汇总；本模块侧重**学生/课程/选课原始记录计数**

### 成员3（跨院选课写回）
- 选课成功后，可通过本模块 `POST /api/drop` 撤销跨院选课记录
- 退选只需 `target_college`（课程所属学院），与选课时的 `target_college` 一致

### 成员5（集成端 GUI）
- Tab 1 / Tab 5 / Tab 6 已适配新 API 格式，可直接使用
- `parse_statistics_response()` 可供其他前端复用

### 成员6（C 系统联调）
- C 学院 API 服务器需提供以下端点，统计模块才能完整统计 C 学院数据：
  - `GET /c/students`
  - `GET /c/courses`
  - `GET /c/selections/xml`
  - `POST /c/drop`（退选模块依赖）
- 若 C 学院离线，统计接口仍正常返回，C 的 `details.C` 中 `online: false`

## 十、技术决策说明

| 决策 | 选择 | 原因 |
|------|------|------|
| 退选实现方式 | 单步转发至目标学院 | 与选课模块对称；集成层无本地数据库，无需同步删除 |
| 统计数据来源 | 拉取 XML 后计数 | 与项目 XML 集成架构一致；不依赖各学院 `/statistics` JSON 接口，数据更权威 |
| 选课计数 | `count_enrollments_xml()` 多标签兼容 | 三学院选课 XML 标签各异（choice/selection/enrollment），统一层处理差异 |
| 离线容错 | 单学院失败 skip + continue | 满足「某学院离线不导致整个统计失败」的需求 |
| 统计响应结构 | 扁平汇总 + details 字典 | 便于按学院 ID 直接索引；GUI 层再转为列表展示 |
| 异常处理 | 双层 try/except（学院级 + 全局级） | 学院级保证互不影响；全局级保证接口始终有结构化响应 |
| API 文档 | 同步更新 `api_docs.md` | 与成员1文档风格一致，供联调与验收参考 |
| GUI 适配 | `parse_statistics_response()` 统一解析 | Tab 1 与 Tab 6 共用，避免重复解析逻辑 |
